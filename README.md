# AAR-Prompt
自动生成AAR的提示词，给DeepSeek：  
我是华为终端手机软件部的软件开发工程师，帮我构造10条复盘总结，每条复盘总结包括：1.标题 2.我们原来期望发生什么？ 3.实际发生了什么？ 4.超出期望或造成差异的原因是什么？我们从中学到了什么？ 5.下一次我们该怎么做？

注意：每个部分不要互相重复，回答包含分析过程、数据支撑、偏差分析等详细信息，所有回答的内容都与复盘标题关联。

一条复盘答案示例：  
标题
```text
注意鸿蒙系统沙箱路径转换
```
我们原来期望发生什么？
```text
在处理文件拉取需求时，我们的预期是：
应用通过日志输出文件存储位置（如/data/storage/el2/base/haps/entry/logs/app.log），开发/测试人员直接复制该路径，即可通过HDC命令成功拉取文件。
认为HDC作为官方调试工具，应能“理解”沙箱路径的语义，自动完成路径映射转换。
假设应用沙箱路径与HDC可见路径是完全一致的，未对鸿蒙与Android的文件系统差异做专项梳理。
```
实际发生了什么？
```text
错误复现：开发人员从应用日志中复制文件路径/data/storage/el2/base/haps/entry/logs/app.log，执行命令：
hdc file recv /data/storage/el2/base/haps/entry/logs/app.log ./app.log
终端返回错误信息：File not found 或 No such file or directory。
尝试排查：执行hdc shell ls /data/storage/el2/base/haps/entry/logs/，返回结果为空或目录不存在。但应用端代码却明确表示文件已成功写入且无异常抛出。
```
超出期望或造成差异的原因是什么？我们从中学到了什么？
```text
文档阅读不充分（根本原因）：鸿蒙Next的沙箱机制与Android的/data/data/package/路径结构有本质不同。鸿蒙引入了EL（加密层级）、用户ID隔离和HAP（鸿蒙原子化服务）多实例的概念，应用层看到的/data/storage/是经过SA（系统服务）层虚拟化映射后的逻辑视图，而非物理存储的真实布局。
```
下一次我们该怎么做？
```text
必须建立“鸿蒙专属排查思维”：不能再依赖Android的经验来推演鸿蒙行为。特别是文件系统、权限管理、组件生命周期等底层能力，二者存在显著差异，必须回归官方文档和源码级说明。
工具链认知需要体系化：HDC不仅仅是adb的简单替换，其子命令（如file recv/send、shell、install）的行为模式各有特点。需要针对HDC专门整理一份《常用命令及路径解析速查表》。
日志输出应兼顾不同视角：应用在打印日志路径时，如果能同时打印逻辑路径和对应的物理路径（如果应用权限允许获取），将极大缩短开发人员的定位时间。
```
